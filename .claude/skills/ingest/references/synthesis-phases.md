# Synthesis Workflow — фазы 1-7

Восемь фаз всего. Phase 8 (lint review) — отдельный документ `references/lint-fix.md`.

Никаких сокращений или импровизации последовательности — каждая фаза имеет свою цель.

---

## Phase 1 — Прочитать источник

Читай **целиком**, без скимминга. Цель — понять структуру знания внутри: какие концепции, какие сущности, какие утверждения, как они связаны.

На этом шаге **никаких страниц не создаётся**. Только понимание.

---

## Phase 2 — Карта знания

Явно (для крупных источников — записать на черновике; для мелких — мысленно) выделить:

```
КОНЦЕПЦИИ В ИСТОЧНИКЕ:
  [концепция 1] — описание, общая или специфичная?
  [концепция 2] — описание
  ...

СУЩНОСТИ В ИСТОЧНИКЕ:
  [сущность 1] — роль в источнике, центральна ли?
  ...

КЛЮЧЕВЫЕ УТВЕРЖДЕНИЯ:
  [утверждение 1] — потенциальные противоречия с существующими страницами wiki?
  ...
```

Этот шаг определяет **что** будет в wiki. Без него ingest деградирует в "пересказ источника".

---

## Phase 3 — Решение по гранулярности

### Концепция → idea-страница?

```
Получит ли свою страницу в ideas/?
├── ДА, если ХОТЯ БЫ ОДНО из:
│     • Общая концепция — существует независимо от домена источника
│       (пример: Advantage Function существует в любом RL, не только в RLHF)
│     • Будет переиспользоваться — ≥2 независимых страниц-ссылки
│       (пример: KL-Divergence используется в RLHF, DPO, VAE)
│     • Самодостаточно объяснима — не требует контекста родителя
│
└── НЕТ → остаётся параграфом в наиболее релевантной родительской
          странице. Со ссылкой на общую концепцию, если такая есть.
```

### Сущность → entity-страница?

```
Получит ли свою страницу в entities/?
├── ДА, если ХОТЯ БЫ ОДНО из:
│     • Центральная для понимания источника
│     • Будет упоминаться в других источниках/страницах
│     • Несёт собственную информацию (не просто упомянута мимоходом)
│
└── НЕТ → остаётся упоминанием в тексте без wikilink
```

Для каждой страницы, которая получит создание — **проверить, не существует ли она уже** через Glob/Grep по `wiki/ideas/` или `wiki/entities/`. Если существует — переход к Phase 4 как "обновление".

---

## Phase 4 — Написать или обновить страницы

```
Поиск существующей страницы → есть?
├── ДА → что НОВОГО добавляет источник?
│         • Новый аспект      → добавляем секцию
│         • Уточнение          → обновляем существующее место
│         • Противоречие       → секция "Противоречия" с явной пометкой
│         • Дубль              → пропускаем
│
└── НЕТ → создаём с шаблоном из _templates/
```

### Стиль письма

- **Декларативное настоящее время.** Пиши знание, а не пересказ.
- ❌ "В источнике говорится, что RLHF — это..."
- ✓ "RLHF — это reinforcement learning из человеческой обратной связи. Постановка задачи: ..."

### Шаблоны

- `_templates/idea.md` для `wiki/ideas/`
- `_templates/entity.md` для `wiki/entities/`
- `_templates/question.md` для `wiki/questions/`
- `_templates/domain.md` для `wiki/domains/` (создаются через Phase 7, не на этом шаге)

---

## Phase 5 — Связать страницы

Для каждой созданной/обновлённой страницы заполнить frontmatter и проставить inline-ссылки.

### Frontmatter

- `summary: '...'` — **обязательно для idea/entity/domain/question.** Одна строка ≤120 символов в одинарных кавычках (YAML-safe для `:`, `$`, `\`). Это то, что попадёт в `wiki/index.md` (генерируется автоматически). Пиши декларативно: «Что это и зачем», без отсылок к источнику. Примеры в существующих страницах.
- `sources: ["[[raw/<file>]]"]` — провенанс на исходный файл (без расширения `.md`)
- `related: [...]` — wikilinks на связанные idea/entity страницы из этого ingest
- `tags: [...]` — тематические теги. **Регистр строгий:**
  - аббревиатуры заглавными: `ML`, `RL`, `NLP`, `RLHF`, `LLM`, `LoRA`
  - обычные слова с заглавной первой буквой: `Alignment`, `Optimization`
  - никаких lowercase-аббревиатур (`ml`, `rl`) и никаких смешений
- `status: ...` — **строго одно из**: `evaluation`, `in-progress`, `ready`. Никаких `stable`, `draft`, `done`. Если страница только что создана и содержание не верифицировано — `evaluation`. Для `entity` поле `status` **не ставится** вообще.
- `domain: [...]` — wikilinks на domain-страницы (`wiki/domains/`). **Convention: упорядочивать от частного к общему** — первый в списке считается primary classification (например, для PPO: `[Reinforcement Learning, Machine Learning]`). Это редакторское решение, lint не enforce'ит. Если страница пока без domain — `domain: []`
- `updated: YYYY-MM-DD`

Полная схема — `.claude/skills/wiki/references/frontmatter.md`.

### Inline в тексте

- При первом упоминании концепции/сущности — `[[wikilink]]`
- При повторных упоминаниях в той же странице — обычный текст

### Ссылки на raw-источники — только в frontmatter `sources`

Не упоминай `raw/...` в теле страницы: ни в нарративе, ни в секции "Источники", ни в формулировках типа "согласно raw/RLHF.md". Источник — провенанс, а не часть содержания.

В разделе "Источники" страницы пиши только концептуальную ссылку на entity (`[[Ziegler 2019 — ...]]`), а не на raw-файл. Если хочется указать "это утверждение появилось из X", X должно быть entity или idea-страницей, а не raw.

### Противоречия

На существующей странице:

```markdown
## Противоречия

Конфликт с [[Новый источник]]: эта страница утверждает X, новый источник говорит Y.
Нужно разрешение. Проверь даты, контекст, первоисточники.
```

На новой странице — обратная ссылка.

## Phase 6 — Инфраструктура

### `wiki/index.md`

**Не редактируй вручную.** Файл генерируется автоматически из поля `summary:` во frontmatter каждой страницы скриптом `bin/gen_index.py` (запускается из Stop-hook'а). Твоя задача — заполнить корректное `summary:` во frontmatter созданной/обновлённой страницы (см. Phase 5). Index появится сам.

Саммари должно передавать суть страницы (≤120 символов), чтобы агент в следующих сессиях мог принять решение о релевантности без открытия файла.
```

При добавлении новой страницы — вставить строку в соответствующий раздел. При обновлении существующей (если суть содержательно сменилась) — обновить строку саммари.

Саммари в декларативном настоящем времени, без вступлений ("эта страница о..."). Цель — за 1-2 секунды дать понять, нужно ли открывать страницу.

### `wiki/log.md` — запись СВЕРХУ

```markdown
## YYYY-MM-DD — ingest | <название источника>

- Источник: `raw/articles/filename.md`
- Создано: [[Страница 1]], [[Страница 2]]
- Обновлено: [[Страница 3]] (новая секция X), [[Страница 4]] (уточнение Y)
- Противоречия: [[Страница A]] vs [[Страница B]] по Z
- Ключевой вывод: одно предложение о новом.
```

### `wiki/cache.md`

Перезаписать целиком с учётом нового ingest. Формат — в `.claude/skills/wiki/SKILL.md`.

### `wiki/summary.md`

**Обязательно** обновить счётчики (число страниц по типам, число источников). Активные темы — обновить если изменилась общая картина.

## Phase 7 — Domain proposal

После обновления инфраструктуры — проверить, не созрел ли какой-то тег для создания domain-страницы.

### Правило

```
ЕСЛИ:
  - тег T присутствует в ≥N страницах wiki (по умолчанию N=10)
  - И нет существующей domain-страницы для этого тега
ТО:
  предложить пользователю создать domain-страницу
```

### Шаги

1. Собрать все теги из страниц, созданных или обновлённых в текущем ingest. Это сужает проверку: теги, которые давно зафиксировались и не "тронуты" этим ingest, не пересчитываются.
2. Для каждого тега посчитать общее количество страниц wiki с этим тегом (`grep` по `wiki/ideas/`, `wiki/entities/`, `wiki/questions/`).
3. Для каждого тега, прошедшего порог N=10:
   - Проверить, существует ли domain-страница в `wiki/domains/`, у которой тег в `tags:` или title соответствует.
   - Если не существует — добавить в список предложений.

4. Если список пуст — Phase 7 завершена. Если не пуст — представить пользователю в свободной форме (не multiple choice):

   Один кандидат:
   ```
   📍 Тег `ML` теперь на 12 страницах. Создать domain-страницу
      "Machine Learning" и связать её с этими страницами?
   ```

   Несколько кандидатов:
   ```
   📍 Два тега прошли порог:
      - `ML` (12 страниц) → "Machine Learning"
      - `Alignment` (10 страниц) → "LLM Alignment"

      Создать domain-страницы для них?
   ```

5. Понимать ответ пользователя в свободной форме:
   - Согласие ("да", "ок", "create") → создать с предложенным названием
   - Отказ ("нет", "skip") → пропустить (без записи отказа)
   - Уточнение названия ("назови AI/ML") → создать с указанным
   - Частичное при множественных ("только первый") → выполнить указанные

### Создание domain-страницы при согласии

1. Создать `wiki/domains/<Title>.md` из шаблона `_templates/domain.md`.
2. Заполнить:
   - Краткое описание области (1-2 абзаца, опираясь на содержимое связанных страниц)
   - Раздел "Ключевые концепции" — 3-5 страниц с наибольшим числом backlinks из этой области
   - Bases-запрос (уже есть в шаблоне) — он автоматически подтянет все страницы с `domain contains "[[<Title>]]"`
3. **Обновить все страницы с этим тегом**: добавить в frontmatter `domain: ["[[<Title>]]"]`. Если поле уже есть с другими значениями — дописать в список **в правильной позиции**.

   **Convention: домены в `domain:` упорядочены от частного к общему.** Первый в списке — primary classification (используется для раскраски на knowledge map). Например, для страницы про PPO правильный порядок:
   ```yaml
   domain:
     - "[[Reinforcement Learning]]"   # узкий
     - "[[Machine Learning]]"          # широкий
   ```
   Это редакторское решение, не enforced (см. `wiki/references/frontmatter.md`). При вставке нового домена ставь его в позицию по убыванию специфичности.
4. Записать в `wiki/log.md` сверху:
   ```markdown
   ## YYYY-MM-DD — domain | <Title>

   - Создано: [[domains/<Title>]]
   - Связано страниц: N (по тегу `<тег>`)
   - Триггер: ingest <источник>
   ```
5. Обновить `wiki/index.md` (добавить ссылку в секцию Domains).
6. Обновить `wiki/cache.md` (упомянуть создание domain).

### При отказе

Ничего не сохранять, никаких skip-листов. Если порог снова будет пройден на следующем ingest — предложение повторится. Пользователь сам решает, когда домен созрел.
