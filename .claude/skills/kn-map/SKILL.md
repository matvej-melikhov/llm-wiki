# kn-map: визуализация wiki через UMAP

Скилл — тонкая обёртка над `bin/knowledge_map.py`. Сам Python-скрипт делает всю работу: читает `wiki/meta/embeddings.json`, проецирует 4096-мерные эмбеддинги в 2D через UMAP, рисует Plotly-фигуру, считает статистику, рендерит markdown-обёртку.

Скилл нужен только для: (1) маршрутизации запроса к скрипту, (2) preflight-проверки эмбеддингов, (3) подачи результата пользователю.

---

## Команда и флаги

| Команда | Поведение |
|---|---|
| `/kn-map` | Полная карта: HTML + markdown-обёртка с iframe и статистикой. Дефолтный `--seed 42`. |
| `/kn-map --no-edges` | Без wikilink-граней (если граф визуально перегружен). |
| `/kn-map --no-page` | Только HTML-артефакт, без markdown в `wiki/meta/kn-maps/`. Для одноразовой проверки. |
| `/kn-map --seed N` | Другой UMAP random_state — даёт другую укладку точек (используй, если дефолтная плохо разделяет кластеры). |
| `/kn-map --refresh` | Сначала `python3 bin/embed.py update` (пересчёт эмбеддингов изменённых страниц), потом сама карта. **Делать после каждого крупного ingest.** |

Под капотом — `python3 bin/knowledge_map.py [флаги]`. Плюс `--out-dir <DIR>` если нужен другой каталог для HTML (по умолчанию `_attachments/`).

---

## Поток

```
1. (опц., при --refresh) python3 bin/embed.py update    # пересчёт эмбеддингов
2. python3 bin/knowledge_map.py [флаги]                 # генерация
3. Парсинг stdout: пути к артефактам + сводная статистика
4. Сообщение пользователю — пути обоими wikilink'ами + ключевые цифры
```

### Pre-flight: актуальность эмбеддингов

Карта строится на `wiki/meta/embeddings.json`. Если после последнего `bin/embed.py update` были созданы или сильно изменены страницы (типичный сигнал — недавний `ingest`), карта будет **построена на устаревших векторах**: новые страницы либо отсутствуют, либо лежат в старых координатах.

Перед запуском проверь:

1. Существует ли `wiki/meta/embeddings.json`. Если нет — обязательно `python3 bin/embed.py update` сначала, иначе скрипт упадёт.
2. Был ли ingest/правки после последнего update. Если да — рекомендуй пользователю `/kn-map --refresh` или запусти update сам, если флаг передан.

Не блокируй запуск, если эмбеддинги «слегка устарели» — просто упомяни в финальном отчёте, что карта построена без обновления.

## Output для пользователя

После успешного прогона — короткое сообщение:

```
🗺️ Карта обновлена.

- HTML: [[_attachments/knowledge-map-YYYY-MM-DD.html]] (Plotly viz, открой в браузере)
- Markdown-обёртка: [[wiki/meta/kn-maps/knowledge-map-YYYY-MM-DD]] (iframe + статистика, рендерится Obsidian'ом в reading mode)

Сводка: <страниц всего>, <доменов>, <wikilink-граней>, <сирот>.
```

Если в stdout есть строки с warnings (например, страницы без эмбеддинга, fallback на серый цвет) — упомяни их.

## Что скилл не делает

- **Не редактирует content-файлы** (`wiki/ideas/`, `wiki/entities/`). Артефакты пишутся только в `_attachments/` и `wiki/meta/kn-maps/`.
- **Не учитывается в `wiki_hash` lint'а** — папка `wiki/meta/kn-maps/` исключена в `bin/lint.py` (см. lint SKILL, секция Skip-check). Поэтому `/kn-map` не триггерит fresh-audit при следующем `/lint`.
- **Не запускает embedder сам** без флага `--refresh`. Если эмбеддингов нет вообще, упасть и попросить пользователя выполнить `python3 bin/embed.py update` явно — это разовая операция (>1 минута), не автоматизируется без явного сигнала.
- **Не удаляет старые snapshot'ы.** История `knowledge-map-*.md` в `kn-maps/` накапливается специально — это диахронический срез эволюции wiki (как `lint-report-*.md`).

---

## Когда вызывать проактивно

После крупного ingest (≥5 новых страниц) или batch-ingest — упомянуть в финальном отчёте: «можно обновить карту знаний через `/kn-map --refresh`». Не запускать без явного запроса — генерация занимает 10-30 секунд и тратит память на UMAP.

Для иллюстраций ВКР — карта это самый зрелищный артефакт, у пользователя могут быть запросы вида «сделай свежий снимок до защиты». Тогда `--refresh` обязателен.
