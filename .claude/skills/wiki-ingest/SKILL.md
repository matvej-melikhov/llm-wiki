# wiki-ingest: синтез источников в wiki

Прочитать источник → построить карту знания → решить гранулярность → создать или обогатить страницы → связать всё через wikilinks и frontmatter.

Один источник обычно затрагивает 5-15 wiki-страниц. Знание накапливается на одной странице из нескольких источников — это и есть compounding knowledge.

**Стандарт синтаксиса**: пиши весь Obsidian Markdown в правильном Obsidian Flavored Markdown. Wikilinks как `[[Note Name]]`, embeds как `![[file]]`, properties как YAML frontmatter. Для деталей синтаксиса см. скилл `obsidian-markdown`.

---

## Команда и флаги

| Команда | Поведение |
|---|---|
| `/ingest [источник]` | Стандартный режим: прочитать → обсудить ключевые выводы → синтезировать страницы |
| `/ingest --auto [источник]` | Автоматический режим: пропустить обсуждение, сразу синтезировать |
| `/ingest [несколько источников]` | Пакетный режим (см. ниже) |

---

## URL ingestion

Триггер: пользователь передаёт URL начинающийся с `https://`.

Шаги:

1. **Скачай** страницу через WebFetch.
2. **Выведи slug** из пути URL (последний сегмент, lowercase, пробелы → дефисы, без query string).
3. **Сохрани** в `raw/articles/[slug]-[YYYY-MM-DD].md` с заголовком frontmatter:
   ```markdown
   ---
   source_url: [url]
   fetched: [YYYY-MM-DD]
   ---
   ```
4. Продолжи с **Synthesis Workflow** на сохранённом файле.

---

## Image / Vision ingestion

Триггер: пользователь передаёт путь к файлу изображения (`.png`, `.jpg`, `.jpeg`, `.gif`, `.webp`, `.svg`, `.avif`).

Шаги:

1. **Прочитай** файл изображения через инструмент Read. Claude обрабатывает изображения нативно.
2. **Опиши** содержимое: извлеки весь текст (OCR), идентифицируй ключевые идеи, сущности, диаграммы, данные.
3. **Сохрани** описание в `raw/images/[slug]-[YYYY-MM-DD].md`:
   ```markdown
   ---
   source_type: image
   original_file: [исходный путь]
   fetched: YYYY-MM-DD
   ---
   # Изображение: [slug]

   [Полное описание содержимого, транскрибированный текст, видимые сущности и т.д.]
   ```
4. Скопируй изображение в `_attachments/images/[slug].[ext]` если оно не в vault.
5. Продолжи с **Synthesis Workflow** на сохранённом файле описания.

Применение: фотографии whiteboard, скриншоты, диаграммы, инфографика, сканы документов.

## Synthesis Workflow

Шесть фаз. Никаких сокращений или импровизации последовательности — каждая фаза имеет свою цель.

### Phase 1 — Прочитать источник

Читай **целиком**, без скимминга. Цель — понять структуру знания внутри: какие концепции, какие сущности, какие утверждения, как они связаны.

На этом шаге **никаких страниц не создаётся**. Только понимание.

### Phase 2 — Карта знания

Явно (для крупных источников — записать на черновике; для мелких — мысленно) выделить:

```
КОНЦЕПЦИИ В ИСТОЧНИКЕ:
  [концепция 1] — описание, общая или специфичная?
  [концепция 2] — описание
  ...

СУЩНОСТИ В ИСТОЧНИКЕ:
  [сущность 1] — роль в источнике, центральна ли?
  ...

КЛЮЧЕВЫЕ УТВЕРЖДЕНИЯ:
  [утверждение 1] — потенциальные противоречия с существующими страницами wiki?
  ...
```

Этот шаг определяет **что** будет в wiki. Без него ingest деградирует в "пересказ источника".

### Phase 3 — Решение по каждой единице

Для **каждой концепции** применить **критерии гранулярности**:

```
Получит ли свою страницу в ideas/?
├── ДА, если ХОТЯ БЫ ОДНО из:
│     • Общая концепция — существует независимо от домена источника
│       (пример: Advantage Function существует в любом RL, не только в RLHF)
│     • Будет переиспользоваться — ≥2 независимых страниц-ссылки
│       (пример: KL-Divergence используется в RLHF, DPO, VAE)
│     • Самодостаточно объяснима — не требует контекста родителя
│
└── НЕТ → остаётся параграфом в наиболее релевантной родительской
          странице. Со ссылкой на общую концепцию, если такая есть.
```

Для **каждой сущности** применить **критерии создания entity**:

```
Получит ли свою страницу в entities/?
├── ДА, если ХОТЯ БЫ ОДНО из:
│     • Центральная для понимания источника
│     • Будет упоминаться в других источниках/страницах
│     • Несёт собственную информацию (не просто упомянута мимоходом)
│
└── НЕТ → остаётся упоминанием в тексте без wikilink
```

Для каждой страницы, которая получит создание — **проверить, не существует ли она уже** через Glob/Grep по `wiki/ideas/` или `wiki/entities/`. Если существует — переход к фазе 4 как "обновление".

### Phase 4 — Написать или обновить страницы

Для каждой запланированной страницы:

```
Поиск существующей страницы → есть?
├── ДА → что НОВОГО добавляет источник?
│         • Новый аспект      → добавляем секцию
│         • Уточнение          → обновляем существующее место
│         • Противоречие       → секция "Противоречия" с явной пометкой
│         • Дубль              → пропускаем
│
└── НЕТ → создаём с шаблоном из _templates/
```

Стиль письма:
- **Декларативное настоящее время.** Пиши знание, а не пересказ.
- ❌ "В источнике говорится, что RLHF — это..."
- ✓ "RLHF — это reinforcement learning из человеческой обратной связи. Постановка задачи: ..."

Используй шаблоны из `_templates/`:
- `_templates/idea.md` для `wiki/ideas/`
- `_templates/entity.md` для `wiki/entities/`
- `_templates/question.md` для `wiki/questions/`
- `_templates/domain.md` для `wiki/domains/` (создаются отдельным workflow, не на этом шаге)

### Phase 5 — Связать страницы

Для каждой созданной/обновлённой страницы заполнить:

**Frontmatter:**
- `sources: ["[[raw/<file>.md>]]"]` — провенанс на исходный файл
- `related: [...]` — wikilinks на связанные idea/entity страницы из этого ingest
- `tags: [...]` — тематические теги. **Регистр строгий:**
  - аббревиатуры заглавными: `ML`, `RL`, `NLP`, `RLHF`, `LLM`, `LoRA`
  - обычные слова с заглавной первой буквы: `Alignment`, `Optimization`
  - никаких lowercase-аббревиатур (`ml`, `rl`) и никаких смешений
- `status: ...` — **строго одно из**: `evaluation`, `in-progress`, `ready`. Никаких `stable`, `draft`, `done`. Если страница только что создана и содержание не верифицировано — `evaluation`. Для `entity` поле `status` **не ставится** вообще.
- `domain: []` — пока пусто; заполнится при создании domain-страниц
- `updated: YYYY-MM-DD`

**Inline в тексте:**
- При первом упоминании концепции/сущности — `[[wikilink]]`
- При повторных упоминаниях в той же странице — обычный текст

**Ссылки на raw-источники — только в frontmatter `sources`.**
Не упоминай `raw/...` в теле страницы: ни в нарративе, ни в секции "Источники", ни в формулировках типа "согласно raw/RLHF.md". Источник — провенанс, а не часть содержания. В разделе "Источники" страницы пиши только концептуальную ссылку на entity (`[[Ziegler 2019 — ...]]`), а не на raw-файл. Если хочется указать "это утверждение появилось из X", X должно быть entity или idea-страницей, а не raw.

**Противоречия** между страницами помечаются явно:

На существующей странице:
```markdown
## Противоречия

Конфликт с [[Новый источник]]: эта страница утверждает X, новый источник говорит Y.
Нужно разрешение. Проверь даты, контекст, первоисточники.
```

На новой странице — обратная ссылка.

### Phase 6 — Инфраструктура

Обновить служебные файлы:

1. **`wiki/index.md`** — добавить новые страницы в соответствующие секции каталога. **Формат: таблица из двух колонок** — wikilink + краткое саммари (1 строка, ≤120 символов). Саммари передаёт суть страницы, чтобы агент мог принять решение о релевантности без открытия файла.

   Структура файла:

   ```markdown
   ## Ideas

   | Страница | Суть |
   |---|---|
   | [[RLHF]] | Схема: SFT → Reward Model → PPO с KL-якорем к SFT |
   | [[PPO]] | Алгоритм оптимизации policy с per-step KL и clipping |
   | [[Advantage Function]] | $A(s,a) = Q(s,a) - V(s)$ — насколько действие лучше среднего |

   ## Entities

   | Страница | Суть |
   |---|---|
   | [[InstructGPT]] | Модель OpenAI 2022, масштабировала RLHF на GPT-3 |

   ## Domains

   | Страница | Суть |
   |---|---|

   ## Questions

   | Страница | Суть |
   |---|---|
   ```

   При добавлении новой страницы — вставляй строку в соответствующий раздел. При обновлении существующей страницы (если суть содержательно сменилась) — обнови строку саммари.

   Саммари пишется в декларативном настоящем времени, без вступлений ("эта страница о..."). Цель — за 1-2 секунды дать агенту понять, нужно ли открывать страницу.

2. **`wiki/log.md`** — дописать запись СВЕРХУ:
   ```markdown
   ## YYYY-MM-DD — ingest | <название источника>

   - Источник: `raw/articles/filename.md`
   - Создано: [[Страница 1]], [[Страница 2]]
   - Обновлено: [[Страница 3]] (новая секция X), [[Страница 4]] (уточнение Y)
   - Противоречия: [[Страница A]] vs [[Страница B]] по Z
   - Ключевой вывод: одно предложение о новом.
   ```
3. **`wiki/cache.md`** — перезаписать целиком с учётом нового ingest. Формат см. в скилле `wiki`.
4. **`wiki/summary.md`** — обновить **обязательно** счётчики (число страниц по типам, число источников). Активные темы — обновить если изменилась общая картина.

### Phase 7 — Domain proposal

После обновления инфраструктуры — проверить, не созрел ли какой-то тег для создания domain-страницы.

**Правило:**

```
ЕСЛИ:
  - тег T присутствует в ≥N страницах wiki (по умолчанию N=10)
  - И нет существующей domain-страницы для этого тега
ТО:
  предложить пользователю создать domain-страницу
```

**Шаги:**

1. Собрать все теги из страниц, созданных или обновлённых в текущем ingest. Это сужает проверку: теги, которые давно зафиксировались и не "тронуты" этим ingest, не пересчитываются.
2. Для каждого тега посчитать общее количество страниц wiki с этим тегом (`grep` по `wiki/ideas/`, `wiki/entities/`, `wiki/questions/`).
3. Для каждого тега, прошедшего порог N=10:
   - Проверить, существует ли domain-страница в `wiki/domains/`, у которой тег в `tags:` или title соответствует.
   - Если не существует — добавить в список предложений.

4. Если список пуст — Phase 7 завершена. Если не пуст — представить пользователю в свободной форме (не multiple choice):

   Один кандидат:
   ```
   📍 Тег `ML` теперь на 12 страницах. Создать domain-страницу
      "Machine Learning" и связать её с этими страницами?
   ```

   Несколько кандидатов:
   ```
   📍 Два тега прошли порог:
      - `ML` (12 страниц) → "Machine Learning"
      - `Alignment` (10 страниц) → "LLM Alignment"

      Создать domain-страницы для них?
   ```

5. Понимать ответ пользователя в свободной форме:
   - Согласие ("да", "ок", "create") → создать с предложенным названием
   - Отказ ("нет", "skip") → пропустить (без записи отказа)
   - Уточнение названия ("назови AI/ML") → создать с указанным
   - Частичное при множественных ("только первый") → выполнить указанные

**Создание domain-страницы при согласии:**

1. Создать `wiki/domains/<Title>.md` из шаблона `_templates/domain.md`.
2. Заполнить:
   - Краткое описание области (1-2 абзаца, опираясь на содержимое связанных страниц)
   - Раздел "Ключевые концепции" — 3-5 страниц с наибольшим числом backlinks из этой области
   - Bases-запрос (уже есть в шаблоне) — он автоматически подтянет все страницы с `domain contains "[[<Title>]]"`
3. **Обновить все страницы с этим тегом**: добавить в frontmatter `domain: ["[[<Title>]]"]`. Если поле уже есть с другими значениями — дописать в список:
   ```yaml
   domain:
     - "[[Existing Domain]]"
     - "[[<Title>]]"
   ```
4. Записать в `wiki/log.md` сверху:
   ```markdown
   ## YYYY-MM-DD — domain | <Title>

   - Создано: [[domains/<Title>]]
   - Связано страниц: N (по тегу `<тег>`)
   - Триггер: ingest <источник>
   ```
5. Обновить `wiki/index.md` (добавить ссылку в секцию Domains).
6. Обновить `wiki/cache.md` (упомянуть создание domain).

**При отказе:** ничего не сохранять, никаких skip-листов. Если порог снова будет пройден на следующем ingest — предложение повторится. Пользователь сам решает, когда домен созрел.

## Batch Ingest

Триггер: пользователь кладёт несколько файлов или говорит "ingest все".

Шаги:

1. Перечисли все файлы для обработки. Подтверди с пользователем перед стартом.
2. Обработай каждый источник по synthesis workflow. Cross-referencing **между источниками** отложи до шага 3.
3. После всех источников: сделай проход cross-reference. Найди связи между новыми страницами разных источников.
4. Обнови index, cache и log один раз в конце (не на каждый источник).
5. Отчитайся: "Обработано N источников. Создано X страниц, обновлено Y. Вот ключевые связи."

Batch ingest менее интерактивен. Для 30+ источников ожидай значительное время обработки. Чекинься с пользователем после каждых 10 источников.

## Дисциплина токенов

Бюджет токенов важен. Соблюдай эти правила:

- Сначала прочитай `wiki/cache.md`. Если он содержит релевантный контекст — не перечитывай связанные страницы целиком.
- Прочитай `wiki/index.md` для поиска существующих страниц до создания новых.
- Читай не более 3-5 существующих страниц за один ingest. Если нужно 10+ — значит, ты читаешь слишком широко.
- Используй точечные правки (Edit), не Write для всего файла, при обновлении существующих страниц.
- Wiki-страницы держи короткими: 100-300 строк. Если страница больше 300 — раздели её по гранулярности.

## Чего не делать

- **Файлы под `raw/` иммутабельны.** Не модифицируй файлы, которые пользователь туда положил (статьи, транскрипты, изображения).
- **Не пиши страницы как пересказ источника.** Пиши знание (декларативное настоящее время). Источник — провенанс, а не сюжет страницы.
- **Не создавай дублирующие страницы.** Всегда проверяй wiki/index.md и существующие папки перед созданием.
- **Не пропускай Phase 2 (карта знания).** Без неё ingest деградирует в одну большую страницу или в хаос.
- **Не пропускай Phase 5 (связи).** Страница без wikilinks и без `sources:` — мёртвая страница.
- **Не упоминай `raw/...` в теле страницы.** Только `sources` во frontmatter. Дублирование пути источника в нарративе или в секции "Источники" — баг.
- **Не пропускай обновление cache и summary.** Cache — то, что делает следующие сессии быстрыми.
