# wiki-ingest: загрузка источников

Прочитай источник. Напиши wiki. Перекрёстно свяжи всё. Один источник обычно затрагивает 8-15 wiki-страниц.

**Стандарт синтаксиса**: пиши весь Obsidian Markdown в правильном Obsidian Flavored Markdown. Wikilinks как `[[Note Name]]`, embeds как `![[file]]`, properties как YAML frontmatter. Для деталей синтаксиса см. скилл `obsidian-markdown`.

---

## Команда и флаги

| Команда | Поведение |
|---|---|
| `/ingest [источник]` | Стандартный режим: прочитать → обсудить ключевые выводы → создать страницы |
| `/ingest --auto [источник]` | Автоматический режим: пропустить обсуждение, сразу создать страницы |
| `/ingest [несколько источников]` | Пакетный режим (см. ниже) |

---

## URL ingestion

Триггер: пользователь передаёт URL начинающийся с `https://`.

Шаги:

1. **Скачай** страницу через WebFetch.
2. **Выведи slug** из пути URL (последний сегмент, lowercase, пробелы → дефисы, без query string).
3. **Сохрани** в `.raw/articles/[slug]-[YYYY-MM-DD].md` с заголовком frontmatter:
   ```markdown
   ---
   source_url: [url]
   fetched: [YYYY-MM-DD]
   ---
   ```
4. Продолжи с **Single Source Ingest** начиная с шага 2 (файл теперь в `.raw/`).

---

## Image / Vision ingestion

Триггер: пользователь передаёт путь к файлу изображения (`.png`, `.jpg`, `.jpeg`, `.gif`, `.webp`, `.svg`, `.avif`).

Шаги:

1. **Прочитай** файл изображения через инструмент Read. Claude обрабатывает изображения нативно.
2. **Опиши** содержимое: извлеки весь текст (OCR), идентифицируй ключевые идеи, сущности, диаграммы, данные.
3. **Сохрани** описание в `.raw/images/[slug]-[YYYY-MM-DD].md`:
   ```markdown
   ---
   source_type: image
   original_file: [исходный путь]
   fetched: YYYY-MM-DD
   ---
   # Изображение: [slug]

   [Полное описание содержимого, транскрибированный текст, видимые сущности и т.д.]
   ```
4. Скопируй изображение в `_attachments/images/[slug].[ext]` если оно не в vault.
5. Продолжи с **Single Source Ingest** на сохранённом файле описания.

Применение: фотографии whiteboard, скриншоты, диаграммы, инфографика, сканы документов.

## Single Source Ingest

Триггер: пользователь кладёт файл в `.raw/` или вставляет содержимое.

Шаги:

1. **Прочитай** источник полностью. Никакого скимминга.
2. **Обсуди** ключевые выводы с пользователем. Спроси: "Что подчеркнуть? Насколько детально?" Пропусти этот шаг при флаге `--auto`.
3. **Создай** страницу-саммари в `wiki/sources/`. Используй frontmatter источника из `references/frontmatter.md`.
4. **Создай или обнови** entity-страницы для каждого человека, организации, продукта, репозитория. Одна страница на сущность.
5. **Создай или обнови** idea-страницы для значимых идей и фреймворков.
6. **Обнови** релевантные domain-страницы и их `_index.md`.
7. **Обнови** `wiki/summary.md` если изменилась общая картина.
8. **Обнови** `wiki/index.md`. Добавь записи для всех новых страниц.
9. **Обнови** `wiki/cache.md` контекстом этого ingest.
10. **Допиши** в `wiki/log.md` (новые записи СВЕРХУ):
    ```markdown
    ## [YYYY-MM-DD] ingest | Заголовок источника
    - Источник: `.raw/articles/filename.md`
    - Саммари: [[Заголовок источника]]
    - Создано страниц: [[Страница 1]], [[Страница 2]]
    - Обновлено страниц: [[Страница 3]], [[Страница 4]]
    - Ключевой вывод: одно предложение о новом.
    ```
11. **Проверь противоречия.** Если новая информация конфликтует с существующими страницами, отметь это явно в обеих страницах: добавь раздел "Противоречия" со ссылкой на конфликтующую страницу.

## Batch Ingest

Триггер: пользователь кладёт несколько файлов или говорит "ingest все".

Шаги:

1. Перечисли все файлы для обработки. Подтверди с пользователем перед стартом.
2. Обработай каждый источник по single-ingest флоу. Отложи cross-referencing между источниками до шага 3.
3. После всех источников: сделай проход cross-reference. Найди связи между новыми источниками.
4. Обнови index, cache и log один раз в конце (не на каждый источник).
5. Отчитайся: "Обработано N источников. Создано X страниц, обновлено Y. Вот ключевые связи."

Batch ingest менее интерактивен. Для 30+ источников ожидай значительное время обработки. Чекинься с пользователем после каждых 10 источников.

## Дисциплина токенов

Бюджет токенов важен. Соблюдай эти правила во время ingest:

- Сначала прочитай `wiki/cache.md`. Если он содержит релевантный контекст, не перечитывай полные страницы.
- Прочитай `wiki/index.md` для поиска существующих страниц до создания новых.
- Читай не более 3-5 существующих страниц за один ingest. Если нужно 10+, ты читаешь слишком широко.
- Используй PATCH для точечных правок. Никогда не перечитывай весь файл, чтобы обновить одно поле.
- Wiki-страницы держи короткими. Максимум 100-300 строк. Если страница больше 300 строк, раздели её.

## Противоречия

Когда новая информация противоречит существующей wiki-странице:

На существующей странице добавь раздел:
```markdown
## Противоречия

Конфликт с [[Новый источник]]: эта страница утверждает X, новый источник говорит Y.
Нужно разрешение. Проверь даты, контекст, первоисточники.
```

В саммари нового источника добавь обратную ссылку:
```markdown
## Противоречит

Этот источник говорит Y, но существующая wiki ([[Существующая страница]]) утверждает X.
```

Не перезаписывай старые утверждения молча. Помечай и оставляй пользователю решение.

---

## Чего не делать

- **Файлы под `.raw/` иммутабельны.** Не модифицируй файлы, которые пользователь туда положил (статьи, транскрипты, изображения).
- Не создавай дублирующие страницы. Всегда проверяй индекс и поиск перед созданием.
- Не пропускай запись в log. Каждый ingest должен быть зафиксирован.
- Не пропускай обновление cache. Это то, что делает следующие сессии быстрыми.
