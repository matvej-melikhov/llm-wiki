# LLM Wiki

Obsidian-vault, управляемый агентом. Реализует паттерн LLM Wiki — постоянная, накапливающаяся база знаний.

**Скиллы:** `/ingest`, `/query`, `/lint`, `/save`, `/brainstorm`, `/kn-map`, `/transcribe`, `/defuddle`, `/obsidian-bases`, `/obsidian-markdown`

## Структура wiki

| Путь              | Назначение                                       | Владелец                                                          | Режим агента        |
| ----------------- | ------------------------------------------------ | ----------------------------------------------------------------- | ------------------- |
| `raw/`            | Источники (md, pdf, docx, транскрипты)           | пользователь, `transcribe` (только конвертация из `raw/formats/`) | create              |
| `raw/formats/`    | Бинарные оригиналы (видео/аудио/изображения)     | пользователь                                                      | read-only           |
| `raw/brainstorm/` | Транскрипты brainstorm-сессий                    | `brainstorm`                                                      | create              |
| `raw/meta/*`      | Эмбеддинги источников, dedup-манифест            | `bin/embed.py`, `ingest`                                          | generated           |
| `_attachments/`   | Вложения для wiki-страниц                        | `ingest`                                                          | create              |
| `_templates/`     | Шаблоны Obsidian Templater                       | пользователь                                                      | read-only           |
| `wiki/ideas/`     | Концепции, механизмы, теории                     | `ingest`                                                          | create + edit       |
| `wiki/entities/`  | Люди, статьи, библиотеки, модели                 | `ingest`                                                          | create + edit       |
| `wiki/domains/`   | Навигационные хабы по областям (MOC)             | `ingest` (порог N=10)                                             | create + edit       |
| `wiki/questions/` | Сохранённые ответы из `/save`, `/query`          | `save`, `query`                                                   | create + edit       |
| `wiki/minds/`     | Мысли пользователя (через `/brainstorm`)         | `brainstorm`                                                      | create + edit       |
| `wiki/index.md`   | Каталог всех страниц                             | `bin/gen_index.py`                                                | generated           |
| `wiki/log.md`     | Журнал операций                                  | все скиллы                                                        | **append** (сверху) |
| `wiki/cache.md`   | Горячий кэш ~500 слов (актуальный контекст)      | `ingest`, `save`, `query`, `brainstorm`                           | **overwrite**       |
| `wiki/summary.md` | Обзор vault (счётчики, статус)                   | `ingest`                                                          | overwrite           |
| `wiki/meta/*`     | Эмбеддинги, lint-state, knowledge-maps, дашборды | `bin/*`, `lint`, `gen_dashboards.py`                              | generated           |

**Режимы:**
- **read-only** — агент не пишет (источники, шаблоны, скрипты)
- **create** — создаёт новые файлы, существующие не правит
- **create + edit** — создаёт и точечно правит (Edit, не перезапись)
- **append** — только добавляет, старое не правится
- **overwrite** — перезаписывает целиком при каждом обновлении
- **generated** — derivable артефакт, можно безопасно удалить и пересчитать

## Кэш контекста

`wiki/cache.md` — саммари последнего контекста на ~500 слов. Существует чтобы любая сессия могла получить недавний контекст без полного обхода wiki.

**Обновляй cache.md:**
- После каждого ingest
- После любого значимого обмена через query
- В конце сессии при завершении работы

**Формат:**

```markdown
---
type: meta
title: "Кэш контекста"
updated: YYYY-MM-DDTHH:MM
---

# Недавний контекст

## Последнее обновление

YYYY-MM-DD. [что произошло — 2-4 предложения, что ingest'ировано / о чём query, какие страницы созданы, ключевая связь]

## Ключевые факты

- [[Концепт]] — одно предложение сути, без раскрытия деталей.
- [[Другой концепт]] — формула или ключевое отношение в одну строку.
- ... (6–10 буллетов)

## Недавние изменения

- YYYY-MM-DD: что произошло, какие страницы затронуты ([[wikilink]]).
- ... (3–5 строк, FIFO)
```

## Frontmatter

`_templates/` — источник правды для схем. При создании страницы бери шаблон и заполняй его поля, не добавляя новых без причины.

### Общие правила

1. Каждая wiki-страница начинается с YAML frontmatter.
2. Frontmatter плоский: поля могут быть строками или списками, но не вложенными объектами.
3. Для wiki-страниц используй даты `YYYY-MM-DD`; для `type: meta` допустим datetime.
4. Пустой список пиши как `[]`, непустой — блоком через `-`.
5. Wikilinks в YAML всегда в кавычках: `"[[Страница]]"`.
6. `domain`, `related`, `sources`, `aliases` и `supersedes` — списки, даже если в них один элемент.
7. `sources` ссылается на файлы из `raw/`; эти ссылки не нужно дублировать в теле страницы.

### Общие поля

- `type` — тип страницы. Должен совпадать с шаблоном и расположением файла.
- `summary` — короткая аннотация в одну строку, желательно до 120 символов. Пиши по сути: не "страница описывает X", а "X делает Y".
- `created` — дата создания страницы.
- `updated` — дата последнего содержательного изменения.
- `tags` — тематические теги для фильтрации. Не дублируй `type`. Регистр: аббревиатуры заглавными (`ML`, `RL`, `NLP`, `RLHF`, `LLM`, `LoRA`), обычные слова с заглавной первой буквой (`Alignment`, `Optimization`). Без lowercase-аббревиатур.
- `status` — состояние работы над страницей. Значения зависят от типа.
- `domain` — wikilinks на страницы из `wiki/domains/`, от более узкого домена к более общему. У domain-страницы здесь стоит ссылка на саму себя.
- `related` — важные соседние wiki-страницы, которые помогают читать материал в контексте.
- `sources` — wikilinks на исходники в `raw/`, из которых получена информация.

### Схемы

#### `idea`

Общие поля: `type`, `summary`, `created`, `updated`, `tags`, `status`, `domain`, `related`, `sources`.

Специфичные поля и правила:
- `aliases` — альтернативные названия, аббревиатуры и распространенные варианты написания.
- `status` — `evaluation` для новой или непроверенной идеи, `in-progress` если суть уже описана, но не хватает связей/примеров/источников, `ready` если идея проверена, раскрыта и хорошо связана с wiki.

#### `entity`

Общие поля: `type`, `summary`, `created`, `updated`, `tags`, `domain`, `related`, `sources`.

Специфичные поля и правила:
- `entity_type` — класс объекта: `person`, `organization`, `product`, `repository`, `place`, `paper`, `model`, `dataset` или близкое значение, если точного нет.
- `role` — короткое пояснение роли сущности в wiki: например, "автор статьи", "модель для генерации изображений", "репозиторий фреймворка".
- `status` не используется: сущность фиксирует факт, а не стадию проработки.

#### `question`

Общие поля: `type`, `summary`, `created`, `updated`, `tags`, `status`, `domain`, `related`, `sources`.

Специфичные поля и правила:
- `question` — исходный пользовательский вопрос или его аккуратная формулировка.
- `answer_quality` — качество ответа: `draft`, `solid`, `definitive`.
- `status` — `evaluation` если ответ требует проверки, `in-progress` если ответ полезен, но неполон, `ready` если ответ закрывает вопрос и хорошо связан с источниками/wiki. Не заменяет `answer_quality`.

#### `domain`

Общие поля: `type`, `summary`, `created`, `updated`, `tags`, `status`, `domain`, `related`.

Специфичные правила:
- `domain` — self-reference: domain-страница ссылается на саму себя, чтобы граф и knowledge map видели ее как узел домена.
- `related` — соседние домены или ключевые страницы, которые не входят прямо в этот домен.
- `sources` обычно не используется: domain собирает навигацию по уже созданным страницам.
- `status` — `evaluation` для нового домена, `in-progress` если собраны ключевые страницы, но навигация неполная, `ready` если домен хорошо структурирован и связан.

#### `mind`

Общие поля: `type`, `summary`, `created`, `updated`, `tags`, `status`, `domain`, `related`, `sources`.

Специфичные поля и правила:
- `author` — автор мысли. Обычно `user`.
- `seed` — исходная фраза, вопрос или тезис, из которого выросла заметка.
- `seed_strategy` — способ получения seed, например `explicit` для явно сказанной мысли.
- `status` — `draft` для новой мысли, `stable` для устоявшейся версии, `deprecated` для мысли, замененной новой заметкой через `supersedes`.
- `supersedes` — ссылки на более старые mind-заметки, которые эта заметка заменяет или уточняет.

#### `meta`

Общая схема wiki-страниц не применяется. `meta` использует только служебные поля:
- `title` — отображаемое название служебной страницы.
- `updated` — время обновления служебного артефакта.
- `summary`, `created`, `tags`, `status`, `domain`, `related` и `sources` для meta не требуются.

## Важные правила и ограничения

1. Один концепт = одна страница.
2. Никаких подпапок по темам внутри `ideas/`, `entities/` — структура страниц плоская.
3. Каждая синтезированная страница имеет источник.
4. Каждый ingest/save оставляет запись в `log.md`.
5. Wikilinks — для ссылок между страницами, markdown — для внешних ссылок.
6. `cache.md` — full overwrite-only. Всегда до ~500 слов, если `wc -w wiki/cache.md` >700 слов — сжимай.
7. **`log.md` — append-only.** Новые записи сверху.
8. **Rename/move — только через `bin/rename_wiki_page.py`.** Прямой `mv` или `Write` на новый путь ломают wikilinks.
9. Файлы в `raw/` не редактируются агентом. Создание источников допустимо при `/transcribe` и `brainstorm`.
10. Всегда используй frontmatter при синтезе страниц.
